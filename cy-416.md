# cy-416 智能会议室预约系统

请根据以下要求生成项目代码和 README。

## 项目主要需求

### 1. 项目概述

**智能会议室预约系统（RoomFlow）** 是一款面向企业内部的纯前端会议室管理应用。员工可查看会议室实时状态、预约会议室、管理自己的会议日程；管理员可维护会议室设备与可用时段。所有数据持久化存储在浏览器 localStorage / IndexedDB 中。项目类型：**Web 前端应用**。

### 2. 核心数据模型（≥ 3 个，必须独立成文件）

- **User（用户）**
  - 字段：id, name, email, avatar, department, role（员工/管理员）, created_at
  - 关键关系：一个用户可创建多个 Booking
  - 业务动作：登录（本地模拟）、修改资料
  - **此模型必须独立定义在 `src/models/user.ts`**

- **Room（会议室）**
  - 字段：id, name, floor, capacity, equipment（投影仪/白板/视频会议/电话）, status（可用/维护中/已停用）, images, open_time, close_time
  - 关键关系：被多个 Booking 引用
  - 业务动作：新增会议室、编辑设备、设置维护状态
  - **此模型必须独立定义在 `src/models/room.ts`**

- **Booking（会议预约）**
  - 字段：id, room_id, user_id, title, attendees（数组）, start_time, end_time, status（待开始/进行中/已结束/已取消）, created_at
  - 关键关系：关联 Room 和 User
  - 业务动作：预约会议室、取消预约、时间冲突检测、签到
  - **此模型必须独立定义在 `src/models/booking.ts`**

### 3. 核心页面（≥ 4 个，至少 2 个页面共用同一业务组件）

- **/dashboard（工作台）**
  - 消费模型：Room、Booking
  - 关键交互：今日会议室使用率、我的今日会议、快速预约入口
  - 共享组件：`<RoomStatusCard>`、`<BookingTimeline>`

- **/rooms（会议室列表）**
  - 消费模型：Room、Booking
  - 关键交互：按楼层/容量/设备筛选、查看会议室详情、跳转预约
  - 共享组件：`<RoomStatusCard>`（与 /dashboard 共用）、`<EquipmentTag>`

- **/booking（会议预约）**
  - 消费模型：Room、Booking、User
  - 关键交互：选择会议室、选择时间段、填写参会人、冲突提示
  - 共享组件：`<TimeSlotPicker>`、`<EquipmentTag>`（与 /rooms 共用）

- **/my-bookings（我的会议）**
  - 消费模型：Booking、Room
  - 关键交互：我创建的/我参与的会议列表、取消/编辑、签到
  - 共享组件：`<BookingTimeline>`（与 /dashboard 共用）、`<EmptyState>`

- **/admin/rooms（会议室管理）**
  - 消费模型：Room
  - 关键交互：新增/编辑会议室、维护状态切换
  - 共享组件：`<RoomForm>`、`<EquipmentTag>`（与 /rooms 共用）

### 4. 横切关注点（≥ 2 个）

- **主题切换/暗黑模式**
  - 触达文件层：
    - `stores/themeStore.ts`
    - `constants/themes.ts`
    - 多个组件读取 theme 状态
    - 日历/时间轴组件主题适配

- **全局错误处理/提示**
  - 触达文件层：
    - `utils/message.ts`
    - `components/common/GlobalErrorBoundary.tsx`
    - 所有表单提交和 store 操作调用 message

- **权限控制（前端路由级）**
  - 触达文件层：
    - `router/guards.ts`
    - `stores/authStore.ts`
    - 管理员页面 meta.requiresAdmin

### 5. 共享枚举/常量（≥ 2 组）

- **RoomStatus（会议室状态）**
  - 值：AVAILABLE = 'available', MAINTENANCE = 'maintenance', DISABLED = 'disabled'
  - 定义位置：`src/constants/room.ts`
  - 出现位置：Room 模型、RoomStatusCard、会议室列表、预约页时间轴、admin 管理页、formatters
  - **README 必须列出该枚举的所有出现位置**

- **BookingStatus（预约状态）**
  - 值：UPCOMING = 'upcoming', ONGOING = 'ongoing', ENDED = 'ended', CANCELLED = 'cancelled'
  - 定义位置：`src/constants/booking.ts`
  - 出现位置：Booking 模型、BookingTimeline、我的会议页、冲突检测逻辑、formatters
  - **README 必须列出该枚举的所有出现位置**

### 6. 共享前端组件清单（≥ 3 个）

- `<RoomStatusCard>`：被 /dashboard 和 /rooms 共用
- `<EquipmentTag>`：被 /rooms、/booking、/admin/rooms 共用
- `<BookingTimeline>`：被 /dashboard 和 /my-bookings 共用
- `<TimeSlotPicker>`：被 /booking 使用
- `<RoomForm>`：被 /admin/rooms 使用
- `<EmptyState>`：被多个列表页共用
- `<GlobalErrorBoundary>`：被 App 使用

### 7. 共享 hooks/utils 清单（≥ 2 个）

- `useAuth()`：本地用户登录态与角色判断
- `useLocalStorage()`：localStorage 读写封装
- `useTimeConflict()`：检测预约时间冲突
- `utils/storage.ts`：localStorage / IndexedDB 封装
- `utils/timeRange.ts`：生成时间段、判断会议状态
- `utils/formatters.ts`：状态文本、时间格式化、设备文本

### 8. 状态管理要求

必须使用 Zustand 独立 store：
```
src/stores/
├── authStore.ts
├── roomStore.ts
├── bookingStore.ts
└── themeStore.ts
```

禁止全部用组件内部 state。每个 store 按实体分文件，组件中按需引入。

### 9. 技术要求

- **英文短名（项目标识）**：`roomflow`
- **前端端口**：18416
- **前端框架**：React 18 + TypeScript
- **UI 库**：Ant Design + Tailwind CSS
- **构建工具**：Vite
- **状态管理**：Zustand
- **路由**：React Router 6
- **数据持久化**：localStorage + IndexedDB（使用 idb-keyval）
- **其他依赖**：dayjs、lodash-es

### 10. 文件结构强制清单

```
src/
├── api/              # 本地数据 API 层：userApi.ts, roomApi.ts, bookingApi.ts
├── stores/           # authStore.ts, roomStore.ts, bookingStore.ts, themeStore.ts
├── models/           # user.ts, room.ts, booking.ts（独立数据模型类型定义）
├── types/            # 共享类型补充
├── components/common/# RoomStatusCard, EquipmentTag, BookingTimeline, TimeSlotPicker, RoomForm, EmptyState, GlobalErrorBoundary
├── hooks/            # useAuth.ts, useLocalStorage.ts, useTimeConflict.ts
├── pages/            # Dashboard, Rooms, Booking, MyBookings, AdminRooms
├── router/           # index.tsx + guards.ts
├── utils/            # storage.ts, timeRange.ts, formatters.ts, validators.ts
├── constants/        # room.ts, booking.ts, themes.ts, messages.ts
└── App.tsx
```

**严禁合并职责到单一文件。即使是简单功能，也必须按照上述分层结构拆分，确保后续每一个小改动都至少触达 3-5 个文件。**

### 11. 特殊约束：屎山代码设计（低内聚、高耦合、牵一发动全身）

1. **常量/工具类必须多处耦合**
   - `utils/formatters.ts` 同时包含时间格式化、会议室状态文本、预约状态文本、设备文本、部门文本
   - `constants/messages.ts` 同时包含页面提示文案、表单校验文案、时间冲突文案
   - 修改一个文案可能同时影响多个页面和组件

2. **枚举必须被多处引用**
   - RoomStatus 和 BookingStatus 必须在 models、constants、components、stores、formatters、router guards、时间轴组件中同时存在
   - 新增一个枚举值时，必须修改至少 8 处文件

3. **存储层必须单独管理但全应用引用**
   - `utils/storage.ts` 统一封装 localStorage / IndexedDB
   - 所有 store 和 api 层都必须通过 storage.ts 读写
   - storage.ts 中同时处理数据序列化、版本兼容、冲突检测缓存
   - 修改存储键名或结构需要同步修改所有 api 层

4. **时间冲突检测必须跨多处**
   - 冲突规则在 `utils/timeRange.ts`、`hooks/useTimeConflict.ts`、`bookingStore.ts`、`TimeSlotPicker` 中同时存在
   - 修改判断规则需要同步修改至少 4 处

5. **牵一发动全身的验证标准**
   - 后续如果要求"给 Room 新增一个 'cleaning' 状态"，必须触达：Room 模型、RoomStatus 枚举、roomStore 状态机、RoomStatusCard、会议室列表、预约页时间轴、admin 管理页、formatters、messages、冲突检测逻辑等 ≥ 10 个文件

### 12. 部署要求
